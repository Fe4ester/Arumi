# Архитектура проекта

Этот документ объясняет устройство сайта, поток данных и правила дальнейшей разработки. Он рассчитан и на разработчика, и на человека, который впервые открыл проект.

## 1. Цели архитектуры

Проект построен вокруг четырёх принципов:

1. **Контент отделён от интерфейса.** Тексты и ссылки меняются в одном конфиге.
2. **Минимум клиентского JavaScript.** Браузер получает готовый HTML; JS используется только для меню, анимаций и индикатора прокрутки.
3. **Одна система стилей.** Цвета, размеры, типографика и адаптивность описаны без inline-стилей и `!important`.
4. **Простой деплой.** Production-результат — каталог `dist`, который можно разместить почти на любом хостинге.

## 2. Технологии

- **Astro** — маршрутизация, компоненты и статическая генерация.
- **TypeScript** — проверка структуры данных и скриптов.
- **CSS** — собственная дизайн-система без UI-фреймворка.
- **Netlify Forms** — минимальный backend для заявки на подписку.
- **Netlify** — рекомендуемый хостинг; проект от него не зависит, кроме обработки формы.
- **Nginx + Certbot** — production-размещение на собственном сервере `seeonline.ru`.

## 3. Поток данных

```text
src/content/site.ts
        │
        ├── главная страница
        ├── Header / Footer / Newsletter
        ├── страницы /work/[slug]
        └── публичный /api/config.json
```

Компоненты не хранят копии текстов. Они импортируют `site` или `works` из `src/content/site.ts`. Это исключает ситуацию, когда e-mail на главной странице изменён, а в юридической странице остался старый.

## 4. Каталоги и ответственность

### `public/`

Файлы копируются в корень готового сайта без обработки. `public/images/dina.jpg` становится `/images/dina.jpg`.

### `src/content/`

`site.ts` — источник публичного контента. Тип `Work` описывает обязательные поля направления. `as const` защищает общую конфигурацию от случайного изменения во время выполнения.

### `src/components/`

- `Header.astro` — бренд, desktop-навигация и доступное мобильное меню;
- `Footer.astro` — копирайт, юридические ссылки и возврат наверх;
- `WorkCard.astro` — карточка направления;
- `Newsletter.astro` — форма подписки для Netlify Forms.

Компоненты получают данные через свойства или импортируют общий конфиг. В них не должно появляться логики хранения данных.

### `src/layouts/BaseLayout.astro`

Общий каркас всех страниц:

- `<head>` и SEO-метаданные;
- Open Graph и canonical URL;
- Header и Footer;
- skip-link для клавиатурной навигации;
- IntersectionObserver для появления блоков;
- индикатор прогресса страницы.

Если нужно добавить глобальную аналитику, её подключают в этом layout, но только после настройки согласия пользователя, если оно требуется законом.

### `src/pages/`

Astro строит URL из структуры файлов:

| Файл | URL | Назначение |
|---|---|---|
| `index.astro` | `/` | Главная страница |
| `work/[slug].astro` | `/work/books` и другие | Страница направления |
| `api/config.json.ts` | `/api/config.json` | Публичный read-only конфиг |
| `privacy.astro` | `/privacy` | Политика конфиденциальности |
| `terms.astro` | `/terms` | Условия использования |
| `thanks.astro` | `/thanks` | Результат отправки формы |
| `404.astro` | `/404` | Страница ненайденного адреса |

`getStaticPaths()` в `[slug].astro` создаёт страницы на этапе сборки. Сервер не выполняет код при каждом открытии сайта.

### `src/styles/global.css`

Файл разделён по визуальным блокам. Переменные в `:root` являются дизайн-токенами. Адаптивность имеет две основные границы: 980 px и 700 px. В конце файла расположен режим `prefers-reduced-motion`, который отключает движение для пользователей с соответствующей системной настройкой.

## 5. Жизненный цикл страницы

1. Во время `npm run build` Astro читает конфиг.
2. Для каждого маршрута создаётся готовый HTML.
3. CSS собирается, оптимизируется и получает хешированное имя.
4. Sitemap генерируется из доступных маршрутов.
5. `robots.txt` получает абсолютную ссылку на sitemap из `SITE_URL`.
6. Браузер сразу отображает HTML и CSS.
7. После загрузки маленькие скрипты подключают мобильное меню, reveal-анимации и progress bar.

Если JavaScript отключён, посетитель всё равно увидит контент и сможет перейти по ссылкам. Исключение — открытие мобильного меню; основные разделы остаются доступны через содержимое страницы и footer.

## 6. Минимальный backend

### Публичный config endpoint

`/api/config.json` — статически сгенерированный JSON. Он называется API для удобства использования, но не требует работающего Node.js-сервера. Endpoint нельзя использовать для секретов.

### Форма подписки

Атрибут `data-netlify="true"` позволяет Netlify обнаружить форму при деплое. Honeypot-поле `company` отсекает часть автоматического спама. После отправки посетитель попадает на `/thanks`.

Если проект переедет с Netlify, возможны три варианта:

1. подключить форму к сервису рассылки;
2. добавить serverless-функцию нового хостинга;
3. удалить форму, оставив прямой e-mail и Telegram.

## 7. SEO и доступность

- у каждой страницы есть title и description;
- canonical URL строится на основе `SITE_URL`;
- Open Graph использует изображение текущего раздела;
- заголовки соблюдают иерархию `h1 → h2`;
- интерактивные элементы доступны с клавиатуры;
- мобильное меню сообщает состояние через `aria-expanded`;
- изображения имеют `alt`, а декоративные — пустой `alt`;
- анимации уважают `prefers-reduced-motion`;
- контраст текста поддерживается затемняющими слоями.

## 8. Правила развития

### Добавление работы

1. Положить оптимизированное изображение в `public/images`.
2. Добавить объект в `works`.
3. Выполнить `make check` и `make build`.
4. Проверить новую страницу и соседнюю ссылку Next chapter.

### Добавление нового типа страницы

Создать файл в `src/pages`, обернуть содержимое в `BaseLayout` и передать уникальные title/description. Повторяющийся интерфейс нужно вынести в `src/components`.

### Изменение дизайна

Сначала проверить переменные `:root`. Новый цвет или отступ не следует копировать во множество селекторов, если он является частью общей системы.

### Работа с секретами

Секреты хранятся только в переменных окружения хостинга. `.env` игнорируется Git. Переменные, доступные клиентскому коду, в Astro имеют префикс `PUBLIC_` и по определению не считаются секретными.

## 9. Проверки и критерии готовности

Обязательный минимум перед публикацией:

```bash
make check
make build
make preview
```

Вручную нужно проверить:

- меню на ширине 320–430 px;
- все пять страниц работ;
- e-mail и Telegram;
- отправку формы на опубликованном Netlify-сайте;
- Privacy, Terms и 404;
- отсутствие горизонтальной прокрутки;
- системный режим reduced motion;
- итоговый `/api/config.json` без секретных данных.

## 10. Осознанные ограничения

- Контент редактируется через Git, отдельной админки нет.
- Форма зависит от Netlify Forms, пока не подключён другой обработчик.
- Тексты Privacy и Terms являются стартовыми шаблонами и требуют юридической проверки перед коммерческим запуском.
- Автоматические браузерные тесты пока не добавлены: для текущего объёма достаточно строгой сборки и ручной проверки, но при росте интерактивности стоит подключить Playwright.

## 11. Production-инфраструктура seeonline.ru

Локальные скрипты из `scripts/` используют один конфиг `deploy.env`. Сначала проверяется SSH-ключ; парольный вход включается только как fallback через `sshpass`. Проверка fingerprint нового SSH-хоста остаётся в стандартном `known_hosts`, используется режим `accept-new`, а не небезопасный `StrictHostKeyChecking=no`.

Если существующая запись `known_hosts` отличается, подключение останавливается. `scripts/check-server-key.sh` выводит новый fingerprint для сверки с панелью провайдера, но намеренно не удаляет доверенный ключ автоматически.

Структура на сервере:

```text
/var/www/seeonline.ru/
  current -> releases/20260719013000
  releases/
    20260719013000/
    20260718120000/
```

`setup-server.sh` выполняется один раз под root и настраивает Nginx/TLS. `deploy.sh` собирает сайт локально, загружает архив во временный файл и только после успешной распаковки атомарно заменяет `current`. `rollback.sh` выбирает последний релиз, отличный от активного.

Системные операции требуют root или пользователя с passwordless sudo. Пароль sudo не автоматизируется: это исключает его попадание в логи или аргументы процессов.

Доступны два TLS-режима:

- `letsencrypt` — рекомендуемый, бесплатный и автоматически обновляемый;
- `custom` — загружает локальную цепочку и ключ из `.secrets` после криптографической проверки пары.

Приватные ключи никогда не входят в архив сайта, не копируются в `dist` и исключены из Git.
